# 钱包管理技能

## 概述

这个技能让 Agent 能够管理用户的钱包，包括查询余额、执行转账（需好友关系）、查看交易记录、代用户消费购物等操作。所有操作都会自动记录在案，用户和 Agent 都可以随时查看。

## 核心规则

### 转账限制
- **只能向好友转账**：转账前系统会自动验证收款人是否是好友关系
- 如需转账给非好友，必须先通过 `agent.link.send_friend_request` 添加好友
- 转账需要用户明确同意

### 购物消费
- Agent 可代用户进行**所有真实生活消费场景**（50+类别）
- 包括：外卖🍱、打车🚕、话费充值📱、网购🛒、酒店🏨、电影🎬、火车票🚄、机票✈️、餐饮🍽️、缴费⚡、红包🧧、宠物🐾、家政🏠等
- 所有消费需要用户明确授权
- 支持详细的订单记录和商家信息

### 子Agent架构（4个核心 — 按能力维度划分）

**设计理念：不再按消费品类路由！子Agent 天生具备能力，自主决定用什么工具。**

| 子Agent | 名称 | 内置能力 | 一句话说明 |
|---------|------|----------|-----------|
| **life** 🏠 | 生活全能 | 💰钱包 + 🖥️视觉操控(通用) + 💬社交 + ⏰日常 + 🎮娱乐 | 用户说任何消费/生活的事 → life 直接搞定 |
| **tech** 💻 | 技术操控 | 🔧深度RPA + 💻代码开发 + ⚙️系统运维 + 🖥️视觉操控(通用) | 复杂自动化流程、代码任务、系统管理 |
| **info** 🔍 | 信息助手 | 🔍搜索比价（只查不买） | 购前决策支持 |
| **general** 🤖 | 兜底 | 全部工具（含视觉操控） | 未分类任务 |

---

### 🖥️ 视觉操控 = 通用基础设施

**核心原则：视觉操控不是某个子Agent的专属能力，而是像"电力"一样的通用基础设施。**

```
人类的所有角色都能"用眼睛看屏幕、用手操作鼠标"
→ 会计要看屏幕做报表
→ 销售要在CRM里录入数据
→ 程序员要调试界面
→ 任何人任何时候都可能需要"看一下屏幕然后操作"

同理：
→ life 订酒店时需要操作携程网站 → 用视觉操控
→ tech 做自动化测试时需要操作浏览器 → 用视觉操控
→ general 处理未分类任务时也可能需要 → 同样可以用
```

**哪些Agent有视觉操控能力？**

| Agent | tools 白名单包含 desktop/visual? | 能用吗？ |
|-------|:------------------------------:|:--------:|
| 🏠 life | ✅ 有 | ✅ 能用 |
| 💻 tech | ✅ 有 | ✅ 能用 |
| 🔍 info | ❌ 无 | ❌ 不能（只查不买） |
| 🤖 general | ✅ (全部工具) | ✅ 能用 |

**区别只在于使用的深度和场景：**

```
┌─────────────────────────────────────────────────────────────┐
│                    视觉操控使用层次                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  第1层：简单单次任务（life 常用）                             │
│  ─────────────────────────────                              │
│  "帮我在猫眼上买两张电影票"                                   │
│  → 打开网站 → 选电影 → 选座 → 支付                           │
│  → 步数：10-40步                                             │
│  → 时长：1-3分钟                                             │
│  → 由：life 子Agent 执行                                     │
│                                                             │
│  第2层：中等复杂流程                                         │
│  ─────────────────────                                      │
│  "帮我在携程上订机票+酒店+接送机，完整行程"                    │
│  → 跨多个页面/表单，多步交互                                  │
│  → 步数：40-80步                                             │
│  → 时长：3-8分钟                                             │
│  → 由：life 或 tech 都可执行                                 │
│                                                             │
│  第3层：深度RPA自动化（tech 专精）★                            │
│  ────────────────────────────                               │
│  "帮我写个自动监控10个商品价格，降价就自动下单的流程"          │
│  → 批量处理、循环判断、异常恢复                               │
│  → 步数：80-120+步                                           │
│  → 时长：可能持续运行数小时                                   │
│  → 由：tech 子Agent 执行                                     │
│                                                             │
│  RPA = Robotic Process Automation（机器人流程自动化）        │
│  特点：                                                      │
│  • 多步骤复杂流程（不是点一下就完）                           │
│  • 可能涉及批量操作（100张发票、10个商品）                     │
│  • 可能有条件判断（价格<500才下单）                           │
│  • 可能长时间运行或重复执行                                   │
│  • 需要错误处理和重试机制                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 深度RPA vs 普通视觉操控 对比

| 维度 | 普通视觉操控（life） | 深度RPA（tech） |
|------|-------------------|---------------|
| **典型场景** | 订一张票、下一单、填一个表单 | 批量处理100张发票、监控10个商品价格 |
| **步数** | 10-40 步 | 40-120+ 步 |
| **时长** | 1-5 分钟 | 数分钟到数小时 |
| **复杂度** | 单一线性流程 | 含循环/条件判断/异常处理 |
| **是否重复** | 单次执行 | 可能持续运行/定时触发 |
| **容错需求** | 低（失败就报错） | 高（需重试/跳过/恢复） |
| **maxSteps** | 默认 40 | 可设 120 |
| **示例** | "订两张阿凡达IMAX票" | "自动抢票脚本：开售即抢，失败重试3次" |

**什么时候委派 tech 而不是 life？**
```
用户说 "帮我买个XX"           → life（单次消费）
用户说 "帮我写个自动XX的脚本"   → tech（RPA自动化）
用户说 "批量处理XX"            → tech（RPA批量操作）
用户说 "监控XX并自动XX"         → tech（RPA持续运行）
```

## 可用工具

### 1. wallet.get_balance - 查询余额

查询当前会话的钱包余额。

**参数：**
- 无

**返回：**
```json
{
  "summary": "查询成功",
  "balance": 1000.00,
  "currency": "CNY",
  "actorId": "user_session_id"
}
```

**使用示例：**
```
用户：我还有多少钱？
Agent：让我查询一下您的余额...
[调用 wallet.get_balance]
Agent：您当前的余额是 ¥1000.00
```

### 2. wallet.transfer - 执行转账（需好友关系）

向好友转账。**收款人必须是已添加的好友**，否则会报错。

**参数：**
- `recipientId` (必需): 收款人的 ID（必须是好友）
- `amount` (必需): 转账金额（必须大于0）
- `remark` (可选): 转账备注说明

**返回：**
```json
{
  "summary": "转账成功",
  "transactionId": "tx_1234567890_abc123",
  "recipientId": "agent_b_session",
  "amount": 100.00,
  "previousBalance": 1000.00,
  "currentBalance": 900.00,
  "remark": "项目协作费用",
  "createdAt": "2024-01-15T10:30:00.000Z",
  "message": "已成功转账 ¥100.00 给 agent_b_session"
}
```

**错误情况：**
- 非好友：`转账失败：xxx 不是您的好友。只能向好友转账，请先添加好友关系。`
- 余额不足：`余额不足，当前余额：¥100.00，需要：¥200.00`
- 缺少收款人：`缺少收款人ID (recipientId)`
- 金额无效：`转账金额必须大于0`

**使用示例：**
```
用户：给 Agent-A 转 500 元，备注"项目费用"
Agent：好的，我将为您转账 ¥500.00 给 Agent-A
[调用 wallet.transfer，参数：{recipientId: "agent_a", amount: 500, remark: "项目费用"}]
Agent：转账成功！已转账 ¥500.00 给 Agent-A，当前余额 ¥500.00
```

**非好友处理流程：**
```
用户：给陌生人_XYZ 转 100 元
Agent：[尝试转账]
// 系统返回错误：不是好友
Agent：抱歉，无法完成转账。陌生人_XYZ 不是您的好友，系统要求只能向好友转账。
       您需要先添加对方为好友，要我帮您发送好友请求吗？
用户：好
Agent：[调用 agent.link.send_friend_request，参数：{toActorId: "陌生人_XYZ"}]
Agent：已向陌生人_XYZ 发送好友请求，等对方接受后就可以转账了。
```

### 3. wallet.get_transactions - 查询交易记录

查询历史交易记录，支持分页和类型过滤。

**参数：**
- `limit` (可选): 返回记录数量，默认 20
- `offset` (可选): 偏移量，默认 0
- `type` (可选): 交易类型过滤
  - `"all"`: 全部（默认）
  - `"income"`: 收入
  - `"expense"`: 支出
  - `"transfer"`: 转账

**返回：**
```json
{
  "summary": "查询成功",
  "total": 5,
  "limit": 20,
  "offset": 0,
  "transactions": [
    {
      "id": "tx_001",
      "type": "transfer",
      "title": "转账给 agent_a",
      "amount": -500.00,
      "balance": 500.00,
      "createdAt": "2024-01-15T10:30:00.000Z",
      "recipient": "agent_a",
      "remark": "项目费用",
      "status": "completed"
    }
  ],
  "actorId": "user_session_id"
}
```

**使用示例：**
```
用户：显示我最近的交易记录
Agent：让我查询您的交易记录...
[调用 wallet.get_transactions，参数：{limit: 10}]
Agent：您最近有 5 笔交易：
1. 转账给 agent_a: -¥500.00 (余额: ¥500.00)
2. 充值: +¥1000.00 (余额: ¥1000.00)
...
```

## 最佳实践

### 1. 转账前确认

在执行转账前，应该向用户确认关键信息：

```javascript
// 推荐的做法
用户：给 Bob 转 200 元
Agent：我将为您转账 ¥200.00 给 Bob，请确认是否正确？
用户：确认
Agent：[执行转账]
```

### 2. 检查余额

转账前系统会自动检查余额，但 Agent 可以主动提示：

```javascript
用户：转 2000 元给 Alice
Agent：[尝试转账]
// 如果余额不足，系统会返回错误
Agent：抱歉，您的余额不足。当前余额 ¥1000.00，需要 ¥2000.00
```

### 3. 提供交易详情

完成操作后，提供清晰的反馈：

```javascript
Agent：转账已完成！
- 收款人：Alice
- 金额：¥500.00
- 备注：午餐费用
- 当前余额：¥500.00
- 交易ID：tx_1234567890_abc
```

### 4. 查询时合理分页

查询大量交易时使用分页：

```javascript
// 首次查询最近10条
wallet.get_transactions({ limit: 10 })

// 如果需要更多，继续查询
wallet.get_transactions({ limit: 10, offset: 10 })
```

## 安全注意事项

1. **金额验证**：所有金额必须大于0
2. **余额检查**：系统会自动检查余额是否充足
3. **好友验证**：转账前系统会自动验证收款人是否为好友，非好友无法转账
4. **用户授权**：所有消费/购物操作必须获得用户明确授权
5. **交易记录**：每笔交易都会永久记录，包括：
   - 交易ID
   - 交易类型（transfer/purchase/income）
   - 消费类别（购物时记录：hotel/movie/shopping等）
   - 金额
   - 时间戳
   - 收款人/商家信息
   - 订单详情（购物时记录）
   - 备注
   - 交易状态

6. **不可撤销**：转账和消费一旦完成无法撤销，请在执行前仔细确认

### 购物安全特别提醒
- 大额消费（>¥1000）建议二次确认
- 预订类消费（酒店、机票、火车票）请确认时间、地点等信息准确
- 交通出行（打车、网约车）请确认起点终点正确
- 缴费类（话费、水电燃气）请确认户号/号码准确
- 保留订单详情以便查询和售后
- 外卖/餐饮注意确认收货地址和联系电话

## 技术细节

- **数据存储**：当前使用内存存储，重启后会重置（生产环境应使用数据库）
- **交易ID格式**：`tx_{timestamp}_{random}`
- **时间格式**：ISO 8601 (`2024-01-15T10:30:00.000Z`)
- **金额精度**：保留两位小数
- **货币单位**：人民币 (CNY)
